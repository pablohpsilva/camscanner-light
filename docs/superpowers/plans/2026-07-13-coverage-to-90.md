# Host Test Coverage → ≥90% Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax.

**Goal:** Raise host line coverage of hand-written Dart (excluding generated `*.g.dart`) from **88.76%** to **≥90%**, with real TDD unit/widget tests (and BDD for any net-new user-facing behavior) — no coverage gaming.

**Architecture:** Each task adds one or more NEW test files under `test/` targeting a specific lib file's uncovered lines. New test files are mutually independent (no shared state, no edits to existing files except where noted) → fully parallelizable across subagents. Coverage is measured with `scripts/coverage.sh` (filters generated code per the agreed policy).

**Tech Stack:** Flutter host tests (`flutter test`), `flutter_test`, mock `MethodChannel` via `TestDefaultBinaryMessenger`, `PackageInfo.setMockInitialValues`, in-memory drift (`NativeDatabase.memory()`), existing fakes in `test/support/`.

## Global Constraints

- **Denominator policy (agreed):** coverage = hand-written Dart under `lib/`, EXCLUDING `*.g.dart` / `*.freezed.dart`. Native-only files stay IN the denominator (accepted drag): `native_page_processor.dart`, `ocr/mlkit_ocr_engine.dart`, `feedback/turnstile_widget.dart` — do NOT attempt host tests for these; they are device-verified.
- **Measure with:** `bash scripts/coverage.sh` (full run) or `bash scripts/coverage.sh --no-run` (reuse lcov). Gate: `bash scripts/coverage.sh --gate 90`.
- **TDD:** write the failing test first, watch it fail, then it passes because the code already exists — for coverage top-ups the "red" is the assertion failing if you got behavior wrong; the discipline is: assert real behavior, never `expect(true, true)` or a bare pump with no assertion.
- **Test hygiene:** every test asserts observable behavior/output. No assertion-free tests. No testing of generated code.
- **BDD (host-level, PROVEN):** `build.yaml` now generates `.feature` files under `test/bdd/**` as PLAIN host widget tests (no `IntegrationTestWidgetsFlutterBinding`) that run under `flutter test` and COUNT toward host coverage. For a user-facing coverage gap, prefer a host-BDD scenario in `test/bdd/` — it satisfies the Gherkin mandate AND moves the number. Steps still live in `test/step/` (shared). Use `integration_test/*.feature` (device-gated) ONLY for native-dependent behavior (camera/opencv/mlkit/pdf/print). Pure-logic/wiring/error-branch coverage stays TDD-only.
  - **Gotcha (verified):** the generator strips punctuation when mapping a step phrase to a step filename/function — e.g. `a paper-colored box` → `aPapercoloredBox` / `a_papercolored_box...`. Keep Gherkin step text plain (letters/spaces/digits); make step file + function names match the generator's output exactly. After editing any `.feature` or step, run `dart run build_runner build --delete-conflicting-outputs`, and `git add` ONLY your own new files — revert any incidental reflow of other generated `*_test.dart` (`git checkout -- <file>`).
- **Do not edit** existing test files unless a task explicitly says so. Each task creates NEW `*_test.dart` files → no parallel conflicts.
- **Zero-warning bar:** `flutter analyze` clean on touched files. `dart format` applied.
- Commit messages end with the two trailers used across this repo (Co-Authored-By + Claude-Session).

## Execution protocol (controller)

1. Record BASE commit. Dispatch Wave 1 tasks (1-3) — independent, may run in parallel subagents (each creates distinct new files).
2. After Wave 1 merges, run `bash scripts/coverage.sh` (full). If ≥ 90.5%, STOP — goal met with margin; skip Wave 2.
3. If < 90.5%, dispatch Wave 2 tasks in priority order, re-measuring after each, until `--gate 90` passes with ≥0.5% margin.
4. Final: `bash scripts/coverage.sh --gate 90` green; `flutter analyze` clean; full `flutter test` green except the 2 known-environmental `opencv_edge_detector` failures. Report exact filtered % and pass count.
5. Any BDD scenarios added → run the Android + iOS device gate for those `.feature`s before claiming done (both-platforms rule).
