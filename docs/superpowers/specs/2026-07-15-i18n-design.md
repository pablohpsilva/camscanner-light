# i18n (localization) — design spec

Date: 2026-07-15
Status: approved design, pre-implementation

## Goal

Localize every user-facing string in the CamScanner-light mobile app
(`apps/mobile/`). The app follows the device language when supported, falls
back to English otherwise, and lets the user pick an explicit language (or
"System default") from the Settings screen. The choice persists across
launches.

## Supported languages (11)

| Locale tag | ARB file        | Native name in picker  |
| ---------- | --------------- | ---------------------- |
| `en`       | `app_en.arb`    | English (template)     |
| `pt`       | `app_pt.arb`    | Português (Portugal)   |
| `pt_BR`    | `app_pt_BR.arb` | Português (Brasil)     |
| `es`       | `app_es.arb`    | Español                |
| `fr`       | `app_fr.arb`    | Français               |
| `de`       | `app_de.arb`    | Deutsch                |
| `lb`       | `app_lb.arb`    | Lëtzebuergesch         |
| `tr`       | `app_tr.arb`    | Türkçe                 |
| `ru`       | `app_ru.arb`    | Русский                |
| `zh`       | `app_zh.arb`    | 中文 (Simplified)       |
| `ar`       | `app_ar.arb`    | العربية                |

- Chinese ships as Simplified only, under the bare `zh` tag so every Chinese
  device setting resolves to it.
- Arabic renders RTL automatically (MaterialApp flips `Directionality`).
- Translations are authored by Claude during implementation; ARB files are
  plain JSON so native speakers can refine entries later.

## Decisions already made

1. **Engine:** Flutter's first-party gen-l10n / ARB toolchain (no third-party
   i18n runtime dependency).
2. **Picker includes "System default"** in addition to the 11 languages.
   Fresh installs start on System default.
3. **One ARB file per locale**, keys namespaced by feature
   (`homeDocumentsTitle`, `scanCancel`, `settingsLanguageLabel`, …) and grouped
   under section comments inside each file. No merge tooling.

## Architecture

### 1. gen-l10n setup

- `pubspec.yaml`: add `flutter_localizations` (SDK) and `intl` (version pinned
  by flutter_localizations); set `generate: true` under `flutter:`.
- `l10n.yaml` in `apps/mobile/`:
  - `arb-dir: lib/l10n`
  - `template-arb-file: app_en.arb`
  - `output-localization-file: app_localizations.dart`
  - output under `lib/l10n/gen/` (non-synthetic package, committed like the
    drift `*.g.dart` files).
- `lib/l10n/l10n.dart` exports the generated class plus a
  `extension L10nX on BuildContext { AppLocalizations get l10n; }` helper so
  call sites read `context.l10n.homeDocumentsTitle`.
- The English template carries `@key` description metadata for every entry so
  translators (human or model) have context. Count strings ("3 pages") use ICU
  plural syntax; Russian and Arabic have non-trivial plural categories and must
  provide all required forms.

### 2. Luxembourgish gotcha (must-handle)

Flutter's built-in `GlobalMaterialLocalizations` / `GlobalCupertinoLocalizations`
do **not** support `lb`. Running MaterialApp with locale `lb` without a
fallback asserts at startup. Fix: `lib/l10n/lb_fallback_delegates.dart`
provides Material/Cupertino/Widgets delegates whose `isSupported` accepts
`lb` and whose `load` returns the **German** built-in localizations (app
strings remain Luxembourgish; only framework-internal strings — date pickers,
tooltips, accessibility labels — fall back to German). Unit-tested.

### 3. Locale selection & persistence (mirrors the theme pattern)

- `lib/l10n/locale_store.dart`: `abstract class LocaleStore { Future<Locale?>
  load(); Future<void> save(Locale? locale); }` where `null` = System default.
  `SharedPrefsLocaleStore` persists under key `app_locale` with values
  `system` | `en` | `pt` | `pt_BR` | … (BCP-47-ish underscore tags).
- `lib/l10n/locale_controller.dart`: `LocaleController extends ChangeNotifier`
  with `Locale? localeOverride`, `load()`, `setLocale(Locale?)` — exactly
  parallel to `ThemeController`. Injected through `runCamScannerApp(...)` with
  a production default of `LocaleController(store: SharedPrefsLocaleStore())`
  so tests inject fakes.
- `lib/l10n/locale_resolution.dart`: pure function
  `Locale resolveLocale(List<Locale>? deviceLocales, Locale? override)`:
  1. If `override` != null → override.
  2. Else first device locale that matches a supported locale — exact
     language+country first (`pt_BR`), then language-only (`pt`, `zh`, `ar`
     regardless of device country/script).
  3. Else `en`.

### 4. MaterialApp wiring (`main.dart`)

`CamScannerApp` listens to both controllers (theme + locale). MaterialApp
gains:

- `localizationsDelegates`: `AppLocalizations.localizationsDelegates` + the
  `lb` fallback delegates,
- `supportedLocales: AppLocalizations.supportedLocales`,
- `locale: localeController.localeOverride` (null = follow system),
- `localeListResolutionCallback` delegating to `resolveLocale` so unsupported
  device languages resolve to English (and live device-language changes are
  honored while on System default).

### 5. Settings UI

New "Language" section in `SettingsScreen`, styled like the existing
Appearance section. Tapping it (Key `settings-language`) opens a picker
listing **System default** + 11 languages by native name (Key
`language-option-<tag>`, e.g. `language-option-pt_BR`,
`language-option-system`). Selection applies immediately (controller
notifies, app rebuilds) and persists via the store. The currently active
choice is visibly marked.

### 6. String extraction

Replace every hardcoded user-facing string (~100–120 unique, ~135 `Text(`
call sites) with `context.l10n.<key>` across the 12 screens plus dialogs,
snackbars, menus, tooltips, and semantic labels:

home_screen, page_viewer_screen, donation_screen, settings_screen,
feedback_screen, recognized_text_screen, scan_screen, id_scan_screen,
capture_review_screen, edit_filter_screen, edit_crop_screen,
pdf_preview_screen (+ any shared widgets/dialog helpers they use).

Non-user-facing strings (log messages, exception messages never shown in UI,
keys, file names) are **not** localized. Strings interpolating values use ARB
placeholders, never string concatenation.

## Error handling

- Missing translation key in a locale file → build-time guardrail (see
  testing) fails the host suite; gen-l10n itself falls back to the template at
  runtime, so users would see English rather than a crash.
- Store read failure → controller falls back to System default.
- Unsupported persisted tag (e.g. after a future language removal) → treated
  as System default.

## Testing (TDD + BDD, both platforms — non-negotiable)

**TDD units (written first, red → green):**

- `resolveLocale` matrix: override wins; device pt-BR → pt_BR; device pt-PT →
  pt; device zh-Hant-TW → zh; device ar-EG → ar; unsupported (ja) → en; empty/
  null device list → en; multi-locale device list picks first supported.
- `SharedPrefsLocaleStore` round-trip incl. `system`, unknown-tag tolerance.
- `LocaleController`: load-on-start, notify on set, persists via store.
- `lb` fallback delegates: `isSupported('lb')`, loads without throwing,
  MaterialApp with `locale: lb` pumps successfully in a widget test.
- ARB parity guardrail: a host test parses all `app_*.arb` files and asserts
  every locale has exactly the template's key set (no missing, no orphan
  keys).

**BDD host (`test/bdd/` + shared steps in `test/step/`, generated via
`build_runner`):** `language.feature` scenarios:

1. Device language is Brazilian Portuguese → home screen shows Portuguese.
2. Device language is unsupported (Japanese) → home screen shows English.
3. I change the language to Spanish in Settings → UI updates immediately.
4. The chosen language survives an app relaunch (same storage).
5. Choosing "System default" returns to the device language.

**Device (integration_test, run on a real Android device AND a real iOS
device):** `integration_test/v1_language_settings.feature` (+ generated
test; `v` is unused by the existing lettered features):
change language in Settings on-device, verify persisted after relaunch-
equivalent, verify Arabic renders RTL (`Directionality.of == rtl`).

**Regression:** existing suites keep passing — the test-default locale is
English, so existing `find.text('Documents')`-style finders stay valid, and
steps/tests that assert literals are updated only where a screen's literal
moved to l10n (same English output).

## Parallel decomposition (per project rules)

- **Phase 1 — infra (small, mostly sequential):** pubspec + l10n.yaml +
  English template ARB (all keys, extracted by reading the 12 screens) +
  gen-l10n + `l10n.dart` + locale store/controller/resolution + lb fallback +
  MaterialApp wiring + Settings picker. TDD throughout.
- **Phase 2 — fan-out (independent subagent tasks):**
  - 12 per-screen extraction tasks (replace literals with `context.l10n.*`;
    keys already exist in the template, so tasks don't collide).
  - 10 per-language translation tasks (each writes one `app_<tag>.arb`).
  - BDD feature + steps task; device-test task.
- **Phase 3 — verification:** `flutter analyze`, full host suite, device runs
  on Android and iOS, ARB parity guardrail green. Evidence pasted before any
  "done" claim.

## Non-goals

- No per-document or OCR language selection (ML Kit recognition language is
  untouched).
- No runtime download of language packs; all locales ship in the binary.
- No traditional-Chinese (`zh-Hant`) variant for now.
- The marketing site (`apps/web`) is out of scope.
